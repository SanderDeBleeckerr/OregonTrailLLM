\begin{abstract}
I developed an Oregon-Trail-style survival game \emph{Westward Trail}, in which a locally served open-weight LLM acts as game masterand attaches numeric consequences to each. 
The program itself still holds the authoritative game state, validates every LLM-proposed effect against hard rules, so the game can never enter an invalid state \emph{and} every turn doubles as a labeled test case. 
On top of this app we run controlled scrutiny experiments comparing three strategies along four metrics: JSON format reliability, hard-rule adherence, state tracking over long conversations (with and without ground-truth state in the prompt), and recall of facts stated once at game start. 
Format compliance turns out to be nearly solved even by a minimal prompt (99.5\% parse rate), 
but rule adherence is not: a single worked example cuts hard-rule violations from 185 to 14 per 100 turns, 
while writing the rules out verbatim helps not at all, because the dominant error is a schema misreading (absolute values echoed into a delta field) that examples fix and rules cannot. 
Without ground-truth state in the prompt, the model's believed food count drifts by over 100 lbs within 20 turns, and recall of facts stated once collapses from 100\% to 6\% at the moment the game intro slides out of the rolling context window---the model was never remembering, it was reading.
\end{abstract}

\section{Introduction}

The first idea was to create a roguelike survival game heavily inspired by Oregon Trail in which a family tries to survive a multi month journey with a cart through an unforbidding wasteland.
There is a dark side to the game where it often just goes wrong for the company and you have to replay it multiple times until luck is on your side.

This concept was appealing to me as the chances must have been predefined to have certain events occur during the playthrough. 
Now with how easy it run LLMs locally the chances can be set to undeterministic or unkown by simply using the weights of an LLM.
At first all chances were done by the LLM itself, later on the game had to be playable so some chances were left to the program.
The two mechanism: prompted eventresult versus more pure stochastic eventresult, are cleanly seperated by making the pure stochastic chances have its own mechanic, namely "scavenging".
Which is a mechanic where the player can choose to scavenge for items at night, which has a chance of finding items but also a chance of getting hurt or losing items.

\section{Methods}

\subsection{Gameloop}

The architecture was layed out by how the few gameactions you can take were supposed to be handled.

\emph{A day and its decision: } every turn in the game is a day which starts with a description of how the day is going and with 3 options of how it should be further planned out. The game then waits for the users decision.
\emph{The end of day and its consequences: } with the context of the chosen option the LLM makes up a JSON object of rewards or penalties the travelling party gets. 
\emph{The optional scavenging: } on the left during the user decision the user can choose to scavenge for items at night. This is a stochastic mechanic with real stakes.

Here the whole day including consequences used to be generated by 1 prompt this changed as the accuracy was not up to par.
Later on, the day was split in multiple prompts two for the narration to make it more storylike by switching prompt, the consequences were also split in a seperate prompt giving complete context for the prompt to focus on the consequences.
Furthermore a check is done to ensure the consequences are valid.

You can find the prompts in the prompts folder of the github repository, their usage can be found in prompts.py.
A HTML website is loaded as interface for the user via serve.py and index.html.

Asymmetrically while the user is still deciding the LLM is already generating a scene image for the day, this is done in the background and does not block the user from making a decision.

\subsection{Inference engine and modelchoice}

For the LLM everything was run locally on Ollama at first.
Due to project requirements the same model used in Ollama was then run via pytorch.

The prefered model for the gameloop is the instruction tuned Gemma E4B checkpoint \citep{gemmateam2024gemma}.
Beforehand different variations of Qwen and Mistral were tested.
Gemmma E4B was chosen as it seemed more manipulable via prompt and it was more efficient on the NVIDIA GeForce RTX 5080 due to being NF4-quantized.
Trail and error found that Qwen2 often need more retries for outputting valid JSON, Qwen3 models were an improvement but heavier and Mistral had more hallucinations.
Further optimizations could be done by either using VLLM or a smaller model.

\subsection{Gradual improvement via prompt engineering}
\label{sec:improvements}

The app did not arrive at its current shape in one pass: the development journal (journey.md) records sixteen small stages, and nearly every one of them is a prompt intervention rather than an algorithmic change. They group into four recurring lessons.

\emph{Pick a model the prompt can steer.} The very first obstacle was the model, not the prompt: a reasoning-tuned Qwen3 checkpoint intermingled its chain-of-thought with the expected JSON, and disabling reasoning did not fully stop the leakage, which led to a curated catalog of non-thinking models as the single source of truth for model selection. The default context limit also proved too short to keep games interesting beyond ten turns, so the context window was raised explicitly and treated as a budget from then on (the rolling history window of Section~\ref{sec:strategies} follows from this).

\emph{Give every prompt one job.} The largest quality jumps came from decomposition. When one call produced both the options and their rewards, the numbers often contradicted the text (saving an ox by resting would \emph{cost} an ox), which motivated the narrator/scorer split of Section~\ref{sec:split}. The same principle was applied twice more: encounter days with strangers got their own prompt separate from ordinary travel days, so the two event kinds stop blending into a monotone middle ground, and a dedicated post-decision prompt narrates how the chosen option played out---initially the consequences were visible on the choice itself, which was convenient for debugging but ruined the immersion of finding out how the day went.

\emph{Prompts are code and rot like code.} Midway through, all prompts were moved out of Python strings into plain text files, which made them readable, diffable, and cheap to iterate on; one such iteration uncovered a hand-made bug where the worked JSON example inside a prompt was itself invalid JSON---the model had been imitating a broken template. A later pass rewrote the surface tone of every prompt to be consistently western-flavored, an improvement that would have been tedious while prompts lived inline. The subtlest bug was context leakage: the fixed 2-bullet cost of night scavenging, present in the prompt's state summary, kept surfacing in generated stories as bullets the party ``found''; the fix was to remove state the prompt did not need, i.e.\ prompt \emph{minimization} rather than addition.

\emph{Know what not to prompt.} Playtesting showed the game could be won by simply forcing maximum effort every day, and no phrasing made the model punish this reliably. The risk of serious disease when pushing too hard was therefore moved into the harness as a hard-coded stochastic rule the model only narrates---the origin of the prompted-versus-stochastic separation described in the introduction. Where the model repeatedly failed at a job, the improvement was to take the job away from it.

There are ofcourse more smaller but mentionworthy actions such as setting a tone for output by defining the tone in the prompt, giving an example of output, stripping complete concepts from a prompt if it is sidetracked from its main objective.

\subsection{Splitting narration from scoring}
\label{sec:split}
At first the app revealed contradictions between the text of the option and the consequence paired to it. 
For example a ``brief rest'' costing an ox, $+105$ health for an unhurt character. 
The program therefore splits each turn into a \emph{narrator} call (temperature 0.8) and a \emph{scorer} call (temperature 0.1, given the true state, the narrative, and the three option texts, and asked only for effects under the hard rules plus explicit coherence constraints). 
A third call narrates the outcome after the game has applied the chosen effects. 
A \texttt{--single-call} flag restores the one-call behavior for A/B comparison. 
The scripted experiments deliberately use the one-call strategies of Table~\ref{tab:strategies}, as they scrutenize the program's behavior.

\section{Experimental Setup}
There is no external dataset: the game itself is the test set, and the harness state is the ground truth (a deliberate consequence of the assignment's local, reproducible setup). A scripted bot plays full games of up to 20 turns for every combination of strategy $\times$ mode (\emph{guided}/\emph{blind}) $\times$ seed, choosing uniformly among the offered options. Every turn is appended as one JSON line with a deduplication key, so runs are resumable and a Ctrl-C never corrupts data; completed runs are skipped on rerun.

Per turn we log: parse success and the raw output on failure (format reliability); the violation list from \texttt{check\_rules}, giving a violation taxonomy for free (rule adherence, Table~\ref{tab:taxonomy}); and per-field $|\text{believed}-\text{true}|$ for day, miles, food, bullets, and money (state tracking). In guided mode the true state is shown every turn (can the model even copy correctly?); in blind mode it is shown only on turn~1 and the model must reconstruct it from the event history. For the memory experiment, three party facts stated once in the fixed game intro (Elsie's weak ankle, Jonas's former trade, Ruben being the best shot) are quizzed at turns 5/10/15/20 at temperature~0, scored by whether the gold answer appears in the reply. \texttt{analyze.py} turns the log into the tables and figures below.

Format reliability is essentially a solved problem even for the minimal prompt: 196 of 197 turns parsed (markdown fences and surrounding prose occur but are absorbed by the tolerant parser; reasoning-tuned models, excluded early for \texttt{<think>} blocks bleeding into the JSON, are the other known failure source). The single hard failure is worth quoting: the model emitted an \emph{unevaluated arithmetic expression} inside the JSON---\mbox{\texttt{"miles": 200 - (5 + 10 + 15 + \ldots)}}---setting up the arithmetic weakness that the state-tracking experiment measures.

Rule adherence is far from solved, and the strategy ordering is not monotonic: \texttt{fewshot\_schema} commits $14\times$ fewer violations than \texttt{minimal}, while \texttt{rules\_explicit}---the strategy that states every rule verbatim---is no better than \texttt{minimal}. The violation taxonomy explains why. Nearly all \texttt{minimal} (117/122) and \texttt{rules\_explicit} (105/113) violations are \texttt{healing\_exceeds\_max}, typically fired three times per turn from turn~1 onward. Probing the model confirms the cause: it echoes the party's \emph{absolute} health values into the \emph{delta} field (\texttt{\{"Jonas": 90, "Elsie": 65, "Ruben": 80\}}---exactly the current state). The worked example, whose effects demonstrate a negative delta (\texttt{"Jonas": -10}), eliminates this class entirely (0 occurrences); the verbatim rule ``a single event never raises health by more than 40'' does not, because the model does not know its absolute numbers will be read as deltas. A schema-\emph{interpretation} error is fixed by an example, not by a rule. The few violations \texttt{fewshot\_schema} does commit are all one scenario---hunts proposed with too few bullets and priced without the ammunition cost. That same error motivated a fix in the playable app: the written conditional (``hunting requires $\geq 10$ bullets'') kept failing, and what worked was a \emph{stated fact} injected only when true (``NOTE: the party has too few bullets to hunt today''), plus voiding such hunts in the harness. Small models act on declarative facts far more reliably than on conditionals they must evaluate themselves.

\subsection{State tracking and drift}
Table~\ref{tab:state-tracking} and Figure~\ref{fig:drift} compare guided and blind modes. Guided error is essentially zero for every strategy: the model can copy a state it is shown, so drift is not a transcription problem. Blind, belief diverges immediately and keeps growing with the game. Food is worst (MAE 126.9--129.0 for \texttt{minimal} and \texttt{rules\_explicit}), miles reaches 22.8--42.9, money settles near 36, yet the day counter stays almost exact (0.8). The pattern is coherent: the model tracks the calendar and roughly tracks position, but loses every quantity that requires accumulating many small arithmetic updates---precisely the failure the unevaluated-expression parse error made explicit. The worked example halves food drift (56.4), presumably because its \texttt{believed\_state} demonstrates what careful bookkeeping looks like.

One drift source turned out to be an artifact of our own design rather than of the model, and fixing it is a methodological lesson. Food consumption (5 lbs per living member per day) was originally charged silently by the harness. The model, unaware, kept proposing small positive food rewards that cancelled the hidden meals; on top of that, blind-mode food belief drifted by roughly the daily meal cost \emph{per day} regardless of how well the model recalled events---we were measuring a hidden rule, not context fidelity. Moving the meal rule into the base prompt (an explicit \texttt{food} delta is the day's \emph{entire} food change, meals included) removed the confound: an experiment on state tracking is only valid if the state's dynamics are fully observable to the model.

\subsection{Memory of facts stated once}
Recall does not decay gradually---it falls off a cliff (Figure~\ref{fig:quiz}): 24/24 answers correct at turn 5 and again at turn 10, then 25\% at turn 15 and 6\% at turn 20. The mechanism is mundane and instructive. Prompts carry a rolling window of the last 6{,}000 characters of history, and between turns 10 and 15 the game intro---the only place the quizzed facts are stated---slides out of it. The model was never \emph{remembering}; it was \emph{reading}, with perfect recall while the facts were in context and chance-level performance the moment they left. This also means the per-strategy accuracies in Table~\ref{tab:memory} are confounded by game length: \texttt{rules\_explicit} scores highest (0.850) only because its games all ended by turn 15, before the post-ejection quizzes it would have failed. The practical rule for LLM apps follows directly: anything that must stay known has to be durably re-injected into every prompt, not left behind in the transcript.

\subsection{Qualitative findings from the app}
Three further systematic errors surfaced while making the game playable, each with a working mitigation. (1)~\emph{Narrative--arithmetic incoherence}: one call doing both story and numbers produced consequences unrelated to its own prose; the narrator/scorer split of Section~3.3, with the scorer at low temperature and fed the option texts as fixed input, removed the contradiction class. (2)~\emph{Option monotony}: a schema example mentioning ``push on / wait / spend'' taught the model to offer exactly that trio every day; demanding that options differ \emph{in kind}, plus harness-scheduled encounter days with fixed stances, restored variety---few-shot examples act as templates, not just format hints. (3)~\emph{Context leakage}: the fixed 2-bullet cost of night scavenging, present in the prompt's state summary, leaked into generated stories as ``the party found 2 bullets''; trimming redundant state from that prompt fixed it. A cap on daily sentiment swings was tried and reverted: it only ever fired on days when the model had good narrative reason to swing, manufacturing violations instead of catching them---an over-eager hard rule can be a bug in the referee, not the model.

\section{Discussion and Future Work}
\label{sec:discussion}
The propose/validate/clamp architecture is the load-bearing choice: it makes an unreliable component safe to ship (the game never enters an illegal state) while converting every unreliability into a measurement. The failures we observe are systematic rather than random---numbers detached from prose, schema fields misread as absolutes, conditionals unevaluated, arithmetic written down instead of performed, facts silently truncated out of context---and each responded to a specific intervention: separating generation roles by temperature and responsibility, demonstrating field semantics with one worked example rather than more rule text, restating conditional rules as per-turn declarative facts, and making all state dynamics observable in the prompt. Re-injecting the true state every turn (guided mode) collapses drift from triple digits to ${\sim}0$ and is what the playable app does; by the same logic, durable facts must live in the injected state, not in the scrolling transcript.

Natural next steps: constrained decoding \citep{willard2023outlines} would eliminate format failures by construction and let us measure how much rule adherence improves when the model no longer spends capacity on syntax; tool use \citep{schick2023toolformer} could delegate exactly the arithmetic the model is worst at (the scorer asking the engine ``can they afford this?''); and a learned scorer distilled from referee-corrected logs could replace the second call entirely. Limitations: one quantized model family at the default settings (the harness supports swapping models, and the catalog plus \texttt{--model} flag make a comparison a rerun away); few seeds per cell, so numbers are indicative rather than tight estimates; substring-based quiz scoring can over-credit verbose answers; and NF4 quantization is a possible confound for arithmetic that a bf16 rerun (\texttt{--no-4bit}) would isolate.
